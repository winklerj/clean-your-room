## Phase 19: Dirty Workspace Recovery Completion

- [x] 54. Complete dirty workspace recovery to satisfy WorkspaceMatchesHeadUnlessOwned — spec lines 524-527 require "if the workspace is dirty without a live owner, capture a patch plus changed-file manifest into `state/recovery/{timestamp}/`, set `dirty_snapshot_artifact`, reset the clone to the accepted baseline, and only then allow review, resume, or cleanup." The prior `RecoveryManager.snapshot_dirty_workspace` only wrote `recovery_metadata.json` and never produced a patch, never wrote a manifest, and never reset the clone. Worse, both `reconcile_running_state` and `handle_cancellation` only triggered the snapshot path when `workspace_state != 'clean'` — but the orchestrator never transitioned `workspace_state` to anything else, so in practice the dirty path never fired in production. Two-part fix: (1) `RecoveryManager` now optionally accepts a `CloneManager` (backward-compatible default `None` keeps existing 33 unit tests green); when injected, `snapshot_dirty_workspace` calls `clone_manager.capture_dirty_diff()` to write `patch.diff`, runs `git status --porcelain` (parsed with `split(None, 1)` because `_run_git` strips its stdout, which collapses the leading-space encoding) to write `changed_files.json`, then calls `clone_manager.reset_to_rev(baseline_rev)` to bring the working tree back to the accepted baseline. Capture/reset outcomes are recorded as `patch_captured` / `manifest_captured` / `clone_reset` flags in `recovery_metadata.json` for forensics. (2) Authoritative dirty detection: new `_workspace_appears_dirty()` helper consults `workspace_state` first (cheap) and then asks the injected `CloneManager.is_workspace_clean()` for ground truth, so a clean DB hint is overridden whenever git sees pending changes. Both `reconcile_running_state` and `handle_cancellation` use this helper before snapshotting. (3) `PipelineOrchestrator.__init__` now passes its own `CloneManager` into `RecoveryManager` so the production code path picks up git-based detection automatically (test injection still works). 11 new tests in `tests/test_recovery.py` (`TestSnapshotWithRealClone` ×7: patch diff written, manifest written, clone reset, metadata flags recorded, no-clone-dir fallback, non-git-clone graceful fallback, clean clone produces empty patch; `TestGitDirtyDetection` ×4: reconcile snapshots when DB says clean but git is dirty, cancellation path likewise, clean+clean stays clean, no-clone-manager preserves legacy behavior). 1273 total tests, 0 warnings, lint clean, mypy clean, server boots.

## Phase 21: Kill semantics — terminate live sessions, snapshot, reset, release, mark killed

- [x] 56. Wire `kill_pipeline` through `RecoveryManager.handle_kill` so kill performs the full cascade — spec line 540 requires kill to "Immediately terminate live sessions, snapshot the workspace, reset, release HTN claims, and mark the pipeline killed". Before this task, `PipelineOrchestrator.kill_pipeline` only cancelled the asyncio task and updated `pipelines.status='killed'` — no session termination wait, no workspace snapshot, no reset, no HTN claim release. The cascade existed as `RecoveryManager.handle_cancellation` but was wired only into the cooperative cancel path, so kill silently bypassed the WorkspaceMatchesHeadUnlessOwned safety net. Three-part fix: (1) `RecoveryManager.handle_cancellation` refactored to delegate to a new shared `_terminate_pipeline(pipeline_id, *, terminal_status, force_snapshot)` helper; cancel keeps `force_snapshot=False` (workspace_state hint is authoritative for the cooperative path) and `terminal_status='cancelled'`. (2) New `RecoveryManager.handle_kill(pipeline_id, owner_token=None)` that calls `_terminate_pipeline` with `terminal_status='killed'` and `force_snapshot=True` — kill bypasses the cooperative cancel boundary where workspace_state would normally have been synced, so we must invoke the full snapshot+reset path unconditionally (when a CloneManager is present and the clone path exists). The cascade order matches the cancel path: snapshot dirty workspace → release HTN claims → mark running sessions/stages with terminal_status → mark pipeline killed → write log line + close LogBuffer. (3) `PipelineOrchestrator.__init__` accepts a new `kill_drain_timeout_sec: float = 5.0` parameter; `kill_pipeline` now sets the cancel_event, cancels the asyncio task, awaits with timeout (so a hung SDK adapter cannot indefinitely block kill), and only then calls `recovery_manager.handle_kill`. The method tolerates three production scenarios: an active in-process task (drain → cascade), a hung task that exceeds the timeout (logged and continues to cascade), and the server-restart case where `_active_pipelines` has no entry (skip drain → cascade directly). 14 new tests: `tests/test_recovery.py::TestHandleKill` ×8 (marks pipeline killed, releases HTN claims, marks running sessions killed, marks running stages killed, force-snapshots even when workspace_state='clean' using a real git clone with uncheckpointed work — asserts patch.diff captures the change and the clone resets to baseline, idempotent on already-killed, closes LogBuffer, does not disturb completed stages); `tests/test_orchestrator.py::TestPipelineLifecycle` +5 (kill releases HTN claims end-to-end, kill marks running sessions killed end-to-end, kill drains an active asyncio task to completion before cascading, kill drain timeout does not block — uses a shielded hung task with `kill_drain_timeout_sec=0.1` and asserts return within 2s, kill works without an active task for the server-restart scenario); `tests/test_orchestrator.py::TestKillInvariants` ×1 PBT (Hypothesis parameterizes `n_in_progress` 0–5 × starting_status running/cancel_requested/paused/needs_attention; asserts `COUNT(*) WHERE status='in_progress'` is 0 after kill and `pipelines.status='killed'`; uses `uuid.uuid4().hex[:8]` suffixes for repo/def/claim names so re-runs of the test body do not violate UNIQUE constraints). 1308 total tests (was 1294; +14), 0 warnings, lint clean, mypy clean, server boots.

## Phase 22: Code review checkpoint commits + head_rev advancement after fix turns

- [x] 57. Wire local checkpoint commits + head_rev advancement after every code_review fix turn so each subsequent review round sees the post-fix diff — spec lines 866-867 mandate "After fixing, update `head_rev` and loop back to review the same full diff range". Before this task, `run_code_review_stage` only ever called `_capture_full_diff(clone_path, review_base_rev, head_rev)` with the same `head_rev` it read from the pipeline row at start — fix-agent edits left in the working tree were uncommitted, never advanced `pipelines.head_rev`, and the next review iteration silently re-saw the previous (stale) diff.
